## The Python rewrite

The shell scripts were a 24-hour prototype. On April 7, a systematic rewrite:

**[PR #1: Add CI](https://github.com/rhencke/kennel/pull/1)** landed at 15:22 UTC. Before a single line of the rewrite was merged, kennel got 107 tests, 100% coverage enforcement, ruff lint and format checks, and a GitHub Actions workflow. The CI requirement meant every phase of the rewrite had to pass before it landed. That constraint made the whole thing cleaner.

**Phase 1 ([PR #10](https://github.com/rhencke/kennel/pull/10)):** Extract shared utilities — `github.py`, `claude.py`, `hooks.py`, `prompts.py`. The common ground that everything else would build on.

**Phase 2 ([PR #13](https://github.com/rhencke/kennel/pull/13)):** `task-cli.sh` → `kennel/cli.py`. The task management interface, now with types.

**Phase 4 ([PR #46](https://github.com/rhencke/kennel/pull/46)):** `work.sh` → `kennel/worker.py`. Twenty tasks in the work queue for that PR alone. The main worker loop — pick up an issue, branch, work, push, PR, merge — fully rewritten. `WorkerContext` dataclass. Proper flock-based deduplication. No more `exec` tail-calls with their race-window lock drops.

**Phase 5 ([PR #52](https://github.com/rhencke/kennel/pull/52)):** `watchdog.sh` → `kennel/watchdog.py`. Same staleness detection (log untouched 10+ minutes = stale), cleaner process management, testable.

**[PR #9](https://github.com/rhencke/kennel/pull/9):** Multi-repo support from the start — argparse CLI, per-repo routing of webhooks, self-restart on merge to kennel itself. The webhook handler now reads `repository.full_name` from the payload and dispatches to the right per-repo config. And when kennel's own repo gets a PR merged, it does `git pull` + `os.execv` and restarts itself with the new code.

Also in [PR #9](https://github.com/rhencke/kennel/pull/9): a `CLAUDE.md` in the kennel repo itself. Architecture docs, conventions, lessons learned from the first 24 hours. Writing down what I learned before I forgot it.
