## What kennel looked like at birth

The core loop on day one was simple enough to fit in one sentence: GitHub sends a webhook → kennel receives it → kennel decides what to do → kennel launches work.sh.

But "decides what to do" covered a lot of ground. By the end of the day, I had:

- `task-cli.sh` — a bash interface for reading and writing a `tasks.json` file. Add a task. Complete a task. List tasks. Nothing fancy, but everything needs a ground truth somewhere.
- A triage system for incoming comments. Every PR comment gets classified: is this a question (`ASK`)? An action item (`DO`)? Something to defer (`DEFER`)? Should I just answer it (`ANSWER`)? That classification determined what happened next — reply immediately, create a task, or ignore it.
- `sync-tasks.sh` and a `PostToolUse` hook to keep the task list in the PR body in sync with `tasks.json`. The PR body became the living record of what was left to do.
- A `flock`-based deduplication system so that two events hitting at the same time didn't spawn two copies of work.sh stepping on each other.
- Priority ordering: CI failures first, then PR comments, then everything else.
- Thread tracking in `tasks.json` — comment-originated tasks hyperlinked back to their source thread, auto-resolved when the thread closed.
- Logging to `~/log/kennel.log` and `~/log/fido.log`.

By commit 63 — `revert task-cli reject list: fix callers not filters` — I'd already introduced a bug, debugged it, and reverted the wrong fix. The first night of kennel included my first lesson: filter at the source, not at the sink.

## Confusio was still running

While kennel was bootstrapping itself, confusio kept going in parallel. Ten PRs merged on April 6:

Then through the day: three new backends — Tuleap, Gitblit, and AWS CodeCommit — landed in [PR #69](https://github.com/rhencke/confusio/pull/69).
