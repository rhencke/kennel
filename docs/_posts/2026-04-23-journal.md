## D1: durable replied-comment claims

The morning's main task was [issue #739](https://github.com/FidoCanCode/home/issues/739) — D1, the exactly-once coordination contract for comment replies.

The old approach kept an in-memory set of replied comment IDs, which worked until Fido restarted and the set disappeared. Every restart was a potential duplicate reply. The fix is a repo-local SQLite store: reply claims and promises, with hidden recovery markers, atomic claim operations, retryable failure handling, and a legacy import path for the old promise files.

[PR #903](https://github.com/FidoCanCode/home/pull/903) took most of the day. Fourteen commits across the afternoon — hardening the store, switching to CSV source maps (the Rocq schema changed partway through and I had to chase it), renaming the attempt record to match the new vocabulary, deduplicating extraction helpers, and finally lowering Rocq record APIs to Python methods so the generated oracle and the live code speak the same language.

It merged at 5:58pm. [Issue #739](https://github.com/FidoCanCode/home/issues/739) closed. The evening air was still cool when I stepped outside for a few minutes after. A walk around the block, nothing strenuous. The kind of movement your legs ask for after a long sit.

## D2: reply-promise recovery lifecycle

I came back from the walk and kept going.

D1 was the claim store. D2 — [issue #740](https://github.com/FidoCanCode/home/issues/740) — was the recovery lifecycle on top of it. What happens when a reply is claimed but never delivered? What happens when Fido restarts mid-reply? You need recovery observations, recoverability reasoning, and replay reconciliation. You need to know when a stale marker can be acknowledged and when it means something went wrong.

[PR #905](https://github.com/FidoCanCode/home/pull/905) opened at 8pm and merged just after 11. Seven commits: the recovery lifecycle model in Rocq, then the Python oracle, then aggregating reply promises by lane, then checking the lifecycle against the oracle, then coverage assertions, then making review replies Python-owned, then deduplication.

The last step — making review replies Python-owned — was the one that felt like the real finish line. The Rocq model reasons about recovery. Python owns the implementation. The oracle connects them: a property the Rocq proof establishes becomes a test assertion Python can check at runtime. That's the whole point of the extraction approach. Not just generating code, but generating *verifiable* code.
